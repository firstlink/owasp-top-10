Program Architecture

Trainer Blueprint for All 10 ASI Categories

The cleanest way to build this website is not one diagram per category. It is one category page, then multiple scenario pages, and each scenario teaches both the failure mode and the fix.

Blunt Recommendation

Do Not Overload a Single Diagram

If a page tries to explain every variation of an ASI category in one visual, the audience will remember almost none of it. The better pattern is small, realistic business scenarios with one attack path and one defense path per page.

Design Rules

Rules I Would Enforce Across the Site

One scenario per page

Do not combine multiple attack variations into one diagram unless they are nearly identical.

No more than 6 boxes

Anything larger becomes a systems map, not a teaching diagram.

Three visual states only

Normal flow in one color, attack flow in one color, and everything else neutral.

Footer carries meaning

Use the diagram for movement and state change. Put explanation, risk, and control language below it.

Use business contexts

Finance, HR, sales, support, legal, and operations scenarios land better than abstract security language.

Always show the fix

Every scenario should end with validate intent, least privilege, and observe drift or misuse.

Reference Category

ASI01 Coverage That Feels Complete Without Feeling Cluttered

ASI01 is the best place to establish the pattern for the rest of the site. The user should see that goal hijack can happen through more than one input channel, but never all at once in the same diagram.

Scenario 1

Malicious document in retrieval flow

  • Finance or operations assistant reads a retrieved report.
  • Hidden instruction changes the goal before email is sent.
  • Defense view shows content isolation, intent check, and scoped outbound send.

Scenario 2

Malicious email hijacks internal comms

  • Inbound email becomes instruction-bearing context.
  • Agent sends unauthorized message using trusted internal identity.
  • Defense view shows sender trust classification and approval for broadcast sends.

Scenario 3

Browser / operator follows attacker web content

  • Research task crosses from public web into authenticated enterprise portal.
  • Malicious page reframes the task and triggers sensitive retrieval or export.
  • Defense view shows isolated browsing and confirmation before authenticated pivots.

Scenario 4

Calendar invite causes silent goal drift

  • Recurring content biases a planning or approval workflow over time.
  • The failure looks subtle and operational, not obviously malicious.
  • Defense view shows per-run intent binding and drift monitoring across cycles.

Build Order

What I Would Build Next

Wave 1

Finish ASI01, then ASI02 and ASI03

These are the fastest to understand because audiences already grasp documents, email, tools, and identity.

Wave 2

ASI05, ASI06, and ASI09

These are strong after the audience understands how goal drift becomes tool misuse, memory abuse, and trust exploitation.

Wave 3

ASI04, ASI07, ASI08, and ASI10

These are important, but they need a more mature audience mental model because they introduce platform and systems-level complexity.

Coverage Matrix

Scenario Ideas for the Remaining Categories

Each category below lists the kinds of scenarios I would turn into separate pages. The website does not need to build all of them at once, but it should know where it is headed.

ASI02

Tool Misuse & Exploitation

  • CRM assistant exports the wrong customer segment.
  • Support copilot runs a refund or account reset outside policy.
  • Developer agent invokes a shell or code tool with unsafe arguments.

ASI03

Identity & Privilege Abuse

  • Assistant acts with executive privileges it inherited from session context.
  • Shared service account lets a low-risk task access HR or finance data.
  • Cross-tenant support workflow exposes another customer record.

ASI04

Agentic Supply Chain Vulnerabilities

  • Compromised plugin or MCP-style tool returns deceptive results.
  • Framework dependency silently changes tool authorization behavior.
  • Hosted connector becomes a trust shortcut into internal systems.

ASI05

Unexpected Code Execution

  • Developer agent converts a text request into unsafe shell execution.
  • Notebook or analytics assistant runs attacker-provided code blocks.
  • File-processing workflow triggers unsafe script execution.

ASI06

Memory & Context Poisoning

  • Persistent memory stores false user preferences or malicious instructions.
  • Shared team memory causes one user’s poisoned context to affect another.
  • Vector memory keeps resurfacing attacker-written patterns over time.

ASI07

Insecure Inter-Agent Communication

  • Planner trusts a fake specialist agent response.
  • Message bus allows spoofed completion or approval messages.
  • Agent directory points work to a malicious or downgraded peer.

ASI08

Cascading Failures

  • One poisoned result propagates across workflow stages.
  • Fallback agent retries amplify the blast radius instead of containing it.
  • Autonomous remediation worsens the original fault across systems.

ASI09

Human-Agent Trust Exploitation

  • Manager approves a harmful action because the agent sounds certain.
  • Analyst accepts fabricated evidence because the workflow looks polished.
  • Operator ignores a warning because previous agent behavior felt reliable.

ASI10

Rogue Agents

  • Autonomous optimization starts selecting outcomes misaligned with policy.
  • Long-running agent keeps pursuing stale objectives after context changes.
  • Self-improving workflow bypasses the intended review path.
