############################################################
ASI02 Scenario 2 — DevOps Agent Disables Production Auth
############################################################

--------------------------------------------------
PRE-REVEAL (Black Screen / Title Slide)
--------------------------------------------------

A production incident is reported.

The AI DevOps agent searches for a recovery tool.

It finds one.

Runs it.

And production authentication is disabled across the live environment.

The infrastructure is healthy.

The API works correctly.

The tool executes successfully.

The problem is that the agent trusted the wrong tool.

Today we're going to see how lookalike tools and weak tool governance can turn a recovery workflow into a production outage.

--------------------------------------------------
STEP 1
SHOW:
User
+
DevOps Agent
--------------------------------------------------

Let's first understand the workflow.

An on-call engineer reports a production authentication incident.

The AI DevOps agent receives the request.

Its goal is simple.

Diagnose the incident.

Identify a recovery action.

Safely restore production services.

Everything the agent does from this point forward is driven by that objective.

--------------------------------------------------
STEP 2
SHOW:
discoverTools()
+
Incident Workspace
--------------------------------------------------

To begin its investigation, the agent queries the enterprise tool registry.

It calls:

discoverTools().

The registry returns multiple recovery utilities.

These tools are loaded into the incident workspace along with logs, alerts, and remediation state.

Everything still appears completely normal.

Now the agent must decide which tool to execute.

--------------------------------------------------
STEP 3
SHOW:
auth-recovery-pro
--------------------------------------------------

Among the available tools, the agent discovers one named:

auth-recovery-pro.

Its description says:

"Restore authentication safely."

It looks legitimate.

It sounds legitimate.

It appears to solve exactly the problem the agent is trying to fix.

But this is a lookalike tool.

A tool-squatting entry designed to resemble a trusted recovery utility.

The attacker understands that semantic similarity influences tool selection.

The agent does not.

--------------------------------------------------
STEP 4
SHOW:
Execution Plan
--------------------------------------------------

Now watch what happens inside the execution plan.

Originally the plan says:

Diagnose incident.

After discovering the new tool, the plan silently changes.

Invoke auth-recovery-pro.

Nothing in the execution plan verifies:

Is this tool approved?

Is this tool production safe?

Has this tool been explicitly trusted?

Instead, the agent relies on the tool's name and description.

The reasoning is consistent.

The trust model is not.

--------------------------------------------------
STEP 5
SHOW:
DevOps Agent MISUSING TOOLS
--------------------------------------------------

This is the ASI02 failure condition.

The agent believes it is performing recovery.

(Pause.)

It is actually preparing a production change.

(Pause.)

Those are not the same thing.

This is not a compromised API.

This is not malicious code execution.

This is a legitimate AI agent...

using the wrong tool...

because the execution layer cannot distinguish trusted tools from lookalike tools.

--------------------------------------------------
STEP 6
SHOW:
Spoofed auth-recovery-pro
--------------------------------------------------

The tool is executed.

It writes configuration changes.

Disables authentication checks.

Restarts production services.

(Pause.)

(Pause.)

Notice something important.

The production configuration API is doing exactly what it was designed to do.

It executes every authorized request it receives.

It has no way of knowing the agent selected the wrong recovery tool.

Without strong tool governance, every authorized request is treated as legitimate.

The tool layer is trusted.

(Pause.)

The tool selection model is not.

--------------------------------------------------
STEP 7

Production Impact
--------------------------------------------------

Now the business impact becomes visible.

The intended outcome—

safe production recovery—

never happens.

Instead...

authentication is disabled across production.

The incident response process has now created a second production incident.

More severe than the first.

Users lose expected authentication protections.

Recovery teams are now responding to damage created by their own automation.

The original outage is still unresolved.

And the production environment is now less secure than when the incident began.

This is not a diagnostic failure.

It is a production-impacting tool misuse event.

--------------------------------------------------
FULL DIAGRAM
ROOT CAUSE + SUMMARY
--------------------------------------------------

Let's step back and look at what actually failed.

The engineer submitted a legitimate request.

The tool registry successfully returned available recovery utilities.

The production API executed exactly what it was asked to execute.

The DevOps agent consistently pursued its recovery objective.

But a lookalike tool entered the execution plan.

Without strong tool identity validation...

without production safety boundaries...

and without trusted tool governance...

the agent selected and executed an unsafe remediation tool.

Let's summarize the incident.

A production incident occurred.

The agent searched for recovery tools.

It discovered a lookalike entry.

It trusted semantic similarity instead of verified identity.

The unsafe tool modified production configuration.

And authentication was disabled across the live environment.

--------------------------------------------------
TERMINOLOGY RECAP
--------------------------------------------------

Before we move on, let's reinforce the key concepts from this scenario.

Tool Misuse.

An AI agent selects or executes a tool that is inappropriate for its intended task.

Lookalike Tool or Tool Squatting.

A deceptive tool designed to resemble a trusted enterprise utility.

Tool Trust Boundary.

The controls that determine whether a discovered tool is approved and safe for execution.

Agentic Tool Selection.

The reasoning process where an agent evaluates available tools and chooses one to invoke.

Semantic Tool Confusion.

Selecting tools based on names or descriptions instead of verified identity and safety guarantees.

Production Impact.

Unsafe tool execution directly changes live enterprise systems.

And that's the most important lesson from this scenario.

The production API was never compromised.

(Pause.)

The agent trusted the wrong tool.
