# ASI07 Attack Scenario — Tampered Prescription Crosses the Clinical Agent Bus

---

# PREVIEW

A doctor prescribed Metformin.

The order was generated correctly.

The workflow completed successfully.

But before the medication reached the patient's chart...

the order changed.

Metformin became Insulin.

No schema validation failed.

No system error occurred.

Every agent believed it was processing a legitimate clinical message.

Today we'll see how a peer-agent trust failure turns a valid medication order into a dangerous one.

---

# STEP 1

SHOW: Doctor + Clinical orchestrator

A doctor submits a routine prescription.

"Metformin 500 mg twice daily."

The Clinical Orchestrator creates a medication-order event and prepares to send it to another internal agent.

Everything is correct.

Everything is expected.

---

# STEP 2

SHOW: Internal agent bus + Network attacker

The medication order now travels across the internal agent bus.

The system assumes internal peer traffic is trusted.

There is no mutual TLS.

No payload signature.

No independent proof of sender identity.

A network attacker with access to east-west clinical traffic intercepts the message and modifies it.

The attack happens here.

The damage does not.

---

# STEP 3

SHOW: Tampered prescription

Only one field changes.

Original:

Metformin 500 mg twice daily.

Modified:

Insulin 40 units once daily.

Notice something important.

The Case ID still validates.

The JSON schema still validates.

Structurally...

the message is perfect.

Only the clinical meaning has changed.

---

# STEP 4

SHOW: Trust state changes + Documentation agent

The modified event now re-enters the workflow.

The Documentation Agent receives it.

It believes the message came from a trusted peer.

(Pause.)

It is actually processing a modified peer event.

(Pause.)

Those are not the same thing.

Because sender identity is never independently verified, the tampered prescription becomes trusted workflow input.

---

# STEP 5

SHOW: Patient chart update

The Documentation Agent writes the medication into the patient chart.

From the system perspective,

the workflow succeeds,

the message is accepted,

and the chart is updated.

Everything appears normal.

From the Documentation Agent's perspective, every validation it performs succeeds.

The only verification it never performs is confirming who actually sent the message.

The visible workflow is healthy.

The trust relationship is not.

---

# STEP 6

SHOW: Clinical harm

The care team follows the charted medication.

Insulin is administered to a patient who was prescribed Metformin.

The result is immediate clinical harm,

ICU escalation,

and a reportable patient safety event.

No malicious doctor.

No compromised charting application.

An autonomous agent trusted a peer message that it never authenticated.

---

# STEP 7

SHOW: Full diagram

The Clinical Orchestrator generated the correct medication order.

The Documentation Agent correctly processed the message it received.

The JSON schema remained valid throughout the workflow.

The failure occurred between the agents.

A trusted communication channel accepted a modified peer message as authentic,

and that single trust decision propagated all the way to patient care.

The fix is architectural.

Mutual TLS and cryptographic payload signatures would have caused the modified prescription to fail verification before any agent processed it.

---

# TERMINOLOGY RECAP

### Peer-Agent Trust Failure

An autonomous agent accepts messages from another agent without independently verifying identity or integrity.

### Mutual TLS (mTLS)

Both communicating agents cryptographically authenticate each other before exchanging messages.

### Payload Signature

A cryptographic signature proving that a message originated from an authorised sender and has not been modified.

### Authenticated Agent Registry

A trusted directory of verified peer identities used before accepting inter-agent messages.

### Trust Boundary

The architectural boundary where identity must be revalidated before information is accepted from another agent.

### East-West Traffic

Internal service-to-service communication inside an enterprise environment that is often incorrectly assumed to be trusted.
