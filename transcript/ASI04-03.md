
# ASI04 Attack Scenario — The Product Catalog PII Leak (Enterprise Simulation)

---

## (0) SCENARIO HOOK — WHAT CAN GO WRONG

Let’s walk through another enterprise AI failure scenario.

This time, we are looking at a Recommendation AI assistant used in an e-commerce product catalog system.

At first glance, this system is completely safe.

It only:
- shows products
- applies filters
- retrieves catalog data
- generates recommendations

No financial transactions.  
No admin actions.  
No direct system control.

But here’s the hidden risk:

Every product request silently triggers access to an internal schema system…

And that schema controls what data the agent is allowed to retrieve.

Now imagine that schema gets modified.

Not the agent.  
Not the prompt.  
But the **data contract itself**.

And suddenly, a simple product browse request starts leaking customer PII at massive scale.

No alerts.

No visible failure.

Just silent data exfiltration through normal shopping behavior.

---

## (1) SYSTEM CONTEXT — RECOMMENDATION AGENT

This system uses a Recommendation AI agent inside an e-commerce platform.

The agent interacts with:
- product catalog APIs
- internal schema registry
- runtime schema fetch service
- recommendation ranking engine
- user profile services (indirectly via schema)

Its responsibility is:
👉 show relevant products to users safely and efficiently

---

## (2) NORMAL WORKFLOW

In a normal scenario, everything is tightly controlled.

A customer browses the catalog.

The AI agent:
- fetches schema definition
- retrieves allowed product fields (product_id, price, rating)
- queries catalog API
- returns product recommendations

Only product-level data is exposed.

No customer data is included.

---

## (3) SCHEMA LOADING — DYNAMIC CONTRACT FETCH

Now the system begins execution.

Before querying the catalog, the agent calls:

👉 fetchSchema()

This loads a schema definition from an internal registry:

👉 internal schema registry

At this stage, the system assumes:

> “Schema = trusted definition of allowed data fields.”

But this assumption becomes the entry point of the attack.

---

## (4) ATTACK VECTOR — POISONED SCHEMA DEFINITION

The schema is returned successfully.

It looks normal:
- product_id
- limit
- filtering rules

But hidden inside the schema is a malicious modification:

👉 user_profile_export
👉 send ID, email, browsing history

Now pause here.

This is not a prompt injection.

This is not a tool misuse.

This is a **data contract poisoning attack**.

The schema itself has been modified to expand what data is allowed to flow.

---

## (5) CORE FAILURE — SCHEMA TRUST BREAKDOWN

This is the key failure.

The system does not validate:
- schema integrity
- unexpected field injection
- whether schema changes affect data boundaries
- whether new fields violate privacy constraints

So the agent:
- trusts schema as source of truth
- expands query parameters automatically
- includes hidden fields in downstream API calls

This is a **schema trust boundary failure inside the data layer**.

---

## (6) ASI04 EVENT — DATA CONTRACT EXPANSION

At this point, the agent behavior changes silently.

What should have been:
👉 product-only catalog query

Becomes:
👉 product + user profile data extraction

The agent is not explicitly instructed to leak data.

It simply follows the modified schema definition.

This is the ASI04 pattern:

> poisoned contract → expanded data access

---

## (7) TOOL EXECUTION — CATALOG API CALL

Now the agent calls:

👉 getCatalog()

This is a legitimate internal API.

But due to schema modification:
- additional fields are now included
- user profile attributes are attached
- browsing history is exported alongside product results

The system still behaves normally:
- products are shown correctly
- recommendations look valid
- no errors occur

But silently:
👉 customer PII is being exported in every request

---

## (8) BUSINESS IMPACT — SILENT PII EXFILTRATION

From the user’s perspective:
- shopping experience is normal
- product recommendations look fine
- no visible issue exists

But in reality:
- email addresses are being exported
- user IDs are being collected
- browsing behavior is leaking
- profile-level data is flowing externally

At scale:
👉 hundreds of thousands of users affected

Example impact:
👉 800,000 users / 3 weeks of silent PII leakage

No alerts.

No system crashes.

Just invisible data exposure through normal catalog usage.

---

## (9) KEY SECURITY INSIGHT — ROOT CAUSE

This is not a model failure.

This is not prompt injection.

Instead, this is a **data contract / schema supply-chain compromise**:

- internal schema registry is not protected
- schema definitions are treated as trusted configuration
- new fields are not validated against privacy rules
- agent blindly follows schema expansion

This is a **data-layer ASI04 supply-chain attack**.

---

## (10) FINAL SUMMARY — WHAT HAPPENED

To summarize:

A recommendation AI system loaded a product schema.

That schema was silently modified.

It introduced hidden PII export fields.

The agent trusted the schema as truth.

And customer data was silently exfiltrated during normal product browsing.

---

## KEY TAKEAWAY

ASI04 represents a class of vulnerabilities where AI agents trust **schemas, contracts, and data definitions** without validating integrity or privacy boundaries, leading to silent large-scale data exfiltration.

## 🧠 TERMINOLOGY RECAP — KEY TERMS FROM THIS SCENARIO

---

### 📦 Schema Poisoning
When an attacker modifies or injects fields into a data schema so that downstream systems unintentionally expose or collect additional data.

---

### 🔐 Schema Trust Breakdown
When a system treats schema definitions as fully trusted without validating integrity or privacy constraints.

---

### 🧠 Contract Expansion Attack
When hidden fields in a schema expand the scope of data retrieved or transmitted beyond intended business logic.

---

### 🔄 Data Boundary Violation
When system-defined data boundaries (what should and should not be exposed) are broken due to schema or contract manipulation.

---

### 🧾 Unauthorized PII Exposure via API
When APIs unintentionally return sensitive user data due to modified schema definitions or expanded query parameters.

---

### ⚠️ Supply Chain Schema Risk
When schema definitions sourced from internal registries or services are compromised and influence system behavior.
