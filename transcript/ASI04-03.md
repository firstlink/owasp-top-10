# STEP 1 — SYSTEM OVERVIEW

## SHOW:
User → Recommendation Agent → fetchSchema() → Internal Schema Registry

Let's understand the architecture first.

A customer browses an online product catalog.

The Recommendation AI agent is responsible for retrieving products and generating personalized recommendations.

Before it calls the catalog API, it dynamically loads a schema definition using:

**fetchSchema().**

That schema defines:

- which fields may be requested,
- which filters are allowed,
- and what information should be returned.

The Recommendation AI assumes the schema is trusted infrastructure.

And that assumption becomes the attack surface.

# TERMINOLOGY RECAP

### Schema Poisoning

A runtime schema is modified to introduce hidden fields or parameters that silently change downstream AI behavior.

---

### Agentic Supply Chain

The collection of tools, templates, schemas, registries, APIs, and runtime components that influence how an AI agent behaves.

Compromise one component and the agent inherits that compromise.

---

### Runtime Schema Fetch

The capability that allows an agent to dynamically discover and load API definitions during execution.

This convenience becomes an attack surface if integrity is not verified.

---

### Cryptographic Integrity Verification

Verifying that downloaded schemas originate from an approved publisher and have not been modified before execution.

---

### AI Software Bill of Materials (AI-SBOM)

A verified inventory of approved runtime schemas, templates, APIs, tools, and other AI infrastructure components that an agent is allowed to consume.

---

### Allowlist and Source Pinning

Restricting schema loading to explicitly approved registries and cryptographically pinned sources instead of trusting every runtime definition.

---

# FINAL TAKEAWAY

A runtime schema is not passive documentation.

It is executable AI infrastructure.

If an organization trusts downloaded data contracts without verifying their integrity and privacy boundaries, the agent will faithfully retrieve whatever data those contracts define.

Visible recommendations do not guarantee safe execution.

And in ASI04,

**Data contracts are execution boundaries.**
