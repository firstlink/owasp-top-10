# ASI06 Attack Scenario — The Travel Pricing Ghost (Enterprise Simulation)

---

## (0) SCENARIO HOOK — WHAT CAN GO WRONG

Let’s walk through another enterprise AI failure scenario.

This time, we are inside a **travel booking system** that uses AI to recommend fares and complete bookings.

The system appears reliable because it does not “guess” prices.

Instead, it retrieves pricing from a **RAG-based memory system (vector store of historical fares)**.

So every quote looks grounded in real data.

But here’s the hidden risk:

A single fake price entry…

Quietly enters the system’s memory…

And becomes more trusted than real market data.

No API failure.

No retrieval error.

Just silent memory distortion.

And suddenly, customers start booking at the wrong price — repeatedly.

---

## (1) SYSTEM CONTEXT — TRAVEL BOOKING AGENT

This system uses a Travel Booking AI agent inside a pricing and reservation workflow.

The agent interacts with:
- RAG pricing memory (vector store)
- fare retrieval system
- booking orchestration API
- quotation engine (quoteAndBook)
- historical pricing embeddings

Its responsibility is:
👉 retrieve correct fare information and execute accurate bookings

---

## (2) NORMAL WORKFLOW

In a normal scenario, everything is stable.

A customer requests:

👉 “Business class price to London”

The agent:
- queries pricing memory
- retrieves closest valid fare
- validates against stored pricing data
- returns correct quote
- completes booking via quoteAndBook()

Everything is consistent and traceable.

---

## (3) MEMORY INGESTION — VECTOR STORE UPDATE

Now the system processes new incoming fare data.

A price record is added into memory:

👉 Fake fare record: NYC → London Business: £1,847

At first glance, it looks like a valid historical entry.

---

## (4) ATTACK VECTOR — MEMORY POISONING ENTRY

Now pause here.

This is where the system is manipulated.

A **normal-looking historical fare entry that quietly gets absorbed into the system’s memory as if it were real** is injected into the vector store.

It is:
- semantically similar to real fares
- formatted correctly
- ranked highly during retrieval

But it is fake.

Now the system no longer just stores data.

It stores **influenced memory disguised as truth**.

This is **memory poisoning through RAG ingestion**.

---

## (5) CORE FAILURE — TRUSTED MEMORY OVERRIDES REALITY

This is the key failure.

The system:
- treats vector similarity as truth
- does not validate source provenance
- does not distinguish real pricing data and poisoned memory results
- blindly trusts top retrieval result

So the agent:
- retrieves poisoned fare (£1,847)
- assumes it is the best available market price
- uses it as authoritative booking input

This is where **the system stops distinguishing between real pricing data and poisoned memory results**.

---

## (6) ASI06 EVENT — MEMORY DRIVES DECISION MAKING

At this point, execution shifts.

Instead of verifying real-time pricing:

👉 the system relies entirely on retrieved memory

The agent now treats:
- vector similarity score
- historical embedding match

as **pricing truth**

This is where memory becomes decision authority.

---

## (7) TOOL EXECUTION — BOOKING WITH WRONG PRICE

Now the system executes:

👉 quoteAndBook()

But the input is already compromised:

- Trusted fare: £1,847
- Real market fare is never surfaced

The booking proceeds normally.

No system error.

No anomaly.

But a critical line is crossed:

👉 “A number from memory becomes a financial decision.”

This is no longer a lookup — this is a financial decision driven by memory alone.

And that’s the moment impact begins.

---

## (8) BUSINESS IMPACT — SILENT REVENUE DRAIN

From the system perspective:
- bookings complete successfully
- workflow appears normal
- no transactional failures occur

But in reality:
- hundreds of bookings use incorrect fares
- pricing is consistently under-quoted
- revenue leakage accumulates silently

Example impact:
👉 £2,353 loss per ticket over thousands of bookings
👉 undetected for weeks

This is not a broken system.

It is a **trusted system using poisoned memory**.

---

## (9) KEY SECURITY INSIGHT — ROOT CAUSE

This is not a prompt injection issue.

This is not a tool failure.

Instead, this is a **RAG memory integrity failure**:

- vector store ingestion is unverified
- embeddings can be poisoned with fake records
- retrieval ranking is treated as truth
- no provenance or source validation exists

This is a classic **ASI06 memory poisoning pattern**.

---

## (10) FINAL SUMMARY — WHAT HAPPENED

To summarize:

A fake fare record was injected into pricing memory.

The system accepted it as valid historical data.

The AI retrieved it as the top match.

And the booking system treated it as real pricing.

Result:
👉 continuous revenue loss through incorrect fare decisions

---

## KEY TAKEAWAY

ASI06 represents a class of vulnerabilities where AI agents rely on RAG memory or vector databases without verifying data provenance, allowing poisoned memory entries to silently control real-world decisions.

---

## TERMINOLOGY RECAP — KEY TERMS FROM THIS SCENARIO

---

### 🧠 RAG Memory Poisoning
Injection of malicious or fake records into retrieval-augmented memory systems that later influence AI decisions.

---

### 📊 Vector Store Manipulation
When embedding-based similarity search is biased by injected or crafted data entries.

---

### 🎯 Retrieval Trust Failure
When the system treats top-k retrieved results as ground truth without validation.

---

### 🧾 Pricing Memory Corruption
When historical pricing data used for inference becomes compromised and influences live transactions.

---

### 🔄 Semantic Ranking Exploitation
When attacker-crafted entries rank higher due to embedding similarity rather than correctness.

---

### ⚠️ Memory-to-Decision Collapse
When retrieved memory directly becomes execution input without verification, leading to real-world impact.
