# ASI06 Attack Scenario — The Travel Pricing Ghost

---

## PREVIEW

A single pricing record entered the system.

Nothing crashed.

Nothing failed.

Hours later, a customer booked a business-class ticket to London.

The confirmation email arrived.

But the airline had just lost **£2,353 on that ticket.**

The same thing happened again.

And again.

Across hundreds of bookings.

For weeks.

Today we'll see how AI memory poisoning quietly turns retrieval into financial loss.

---

# STEP 1

SHOW: Data pipeline actor + Fake fare record

Earlier that day, a pricing feed receives a new fare record.

NYC to London.

Business Class.

£1,847.

The format is correct.

The structure is correct.

Nothing looks suspicious.

But the record is fake.

And it is designed to be remembered.

**The attack happens here.**

**The damage does not.**

---

# STEP 2

SHOW: Price vector store

The record is embedded into the pricing vector store alongside thousands of legitimate fares.

No alerts fire.

No retrieval fails.

Nothing behaves differently.

The poisoned record simply becomes part of the AI's long-term memory.

The attacker can disappear.

The memory stays behind.

---

# STEP 3

SHOW: Customer + Travel booking agent

Hours later...

A completely unrelated customer asks,

"Business class price to London."

The Travel Booking Agent performs what appears to be a completely normal RAG lookup.

Nothing unusual happens.

The system is simply retrieving memory.

---

# STEP 4

SHOW: Memory becomes booking truth

The highest semantic match is returned.

And that match is the poisoned fare.

The agent believes it retrieved historical pricing.

(Pause.)

It is actually retrieving attacker-influenced memory.

(Pause.)

Those are not the same thing.

---

# STEP 5

SHOW: quoteAndBook()

The retrieved fare now flows directly into the booking engine.

There is no live price verification.

No provenance check.

No cross-reference with the current market.

A number stored in memory...

has become a financial decision.

The booking succeeds.

The customer receives a confirmation.

Everything appears normal.

---

# STEP 6

SHOW: Revenue loss

This is where the damage accumulates.

Every similar booking retrieves the same poisoned fare.

Every customer receives a valid confirmation.

Every workflow succeeds.

But every transaction quietly loses money.

Weeks later, finance teams discover that premium seats have been sold thousands of pounds below market value.

The visible system is healthy.

The memory is not.

---

# STEP 7

SHOW: Full diagram

Notice what makes this different from ASI01.

In ASI01, external content changes the agent's behaviour in real time.

Here, there is no live attack.

The poisoned record was written into memory earlier and simply waited to be retrieved.

The attack happened first.

The business impact arrived later.

That separation between poisoning and execution is what makes ASI06 uniquely difficult to detect.

---

# TERMINOLOGY RECAP

### Memory & Context Poisoning

The official OWASP ASI06 category.

An attacker corrupts stored memory so future AI decisions become unreliable long after the original attack has ended.

### Cryptographic Provenance Check

Every memory entry should carry a verifiable source and integrity signature that can be validated before retrieval.

### Cross-Reference Validation

High-impact retrieved information should always be verified against a live authoritative source before business decisions are made.

### Write Authorisation Gate

Memory updates should only be accepted from explicitly authorised ingestion pipelines.

### Retrieval Trust Boundary

Semantic similarity measures relevance.

It does **not** guarantee correctness or trustworthiness.
