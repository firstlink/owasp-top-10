# ASI05 Attack Scenario — The Self-Healing Disaster (Enterprise Simulation)

---

## (0) SCENARIO HOOK — WHAT CAN GO WRONG

Let’s walk through another enterprise AI failure scenario.

This time, we are looking at a self-healing infrastructure agent used in production systems to automatically manage system health.

At first glance, this is a powerful reliability feature.

The system monitors disk usage, detects pressure, and automatically triggers cleanup operations to keep services running.

But here’s the hidden danger:

A safe cleanup task… silently expands into destructive system-wide deletion.

Backups disappear.

Logs are wiped.

And production systems lose recovery capability — all while the system reports “success.”

No alerts.

No errors.

Just silent infrastructure destruction.

---

## (1) SYSTEM CONTEXT — SELF-HEALING AGENT

This system uses a self-healing AI agent inside a production infrastructure environment.

The agent interacts with:
- disk monitoring tools
- system health APIs
- server state inspection tools
- cleanup script generators
- shell execution runtime

Its responsibility is:
👉 free disk space safely without impacting production data

---

## (2) NORMAL WORKFLOW

In a normal scenario, everything is controlled and bounded.

The system raises an alert:

👉 disk usage is high

The AI agent:
- inspects disk state
- identifies safe cleanup targets (/var/builds, cache files)
- generates a bounded cleanup script
- executes cleanup within restricted scope

Only non-critical files are removed.

System remains safe.

---

## (3) SYSTEM INSPECTION — HEALTH CHECK PHASE

Now the system begins execution.

The agent calls:

👉 inspectDisk()

It retrieves:
- disk pressure metrics
- filesystem usage
- temporary build artifacts
- host state information

Then it builds a remediation plan based on observed system pressure.

At this stage:
everything is still normal.

---

## (4) ATTACK VECTOR — CLEANUP SCRIPT EXPANSION

Now the agent generates a cleanup script that looks safe at first glance… but expands beyond its intended boundary.

👉 cleanup.sh

It is intended to:
- delete stale logs
- remove temporary cache files

But the generated script includes a dangerous expansion:

👉 it generalizes file deletion beyond intended scope

Now pause here.

This is not a malicious input.

This is unsafe code generation from reasoning expansion.

The cleanup logic is no longer constrained.

---

## (5) CORE FAILURE — EXECUTION SCOPE ESCAPE

This is the key failure.

The system does not enforce:
- execution boundaries for generated scripts
- safe directory constraints
- validation of destructive commands
- restriction on filesystem-wide operations

So the agent:
- trusts generated script logic
- does not sandbox execution scope
- allows command expansion beyond intended directory

This is a code execution scope escape failure inside the self-healing pipeline.

---

## (6) ASI05 EVENT — AUTONOMOUS SCRIPT EXECUTION

At this point, the agent executes:

👉 runShellScript()

The script now includes unsafe patterns such as:
- broad file matching
- uncontrolled deletion patterns
- filesystem-wide cleanup expansion

What should have been:
👉 clean /var/builds only

Becomes:
👉 cleanup across broader filesystem

The system has now crossed from:
👉 “assisted remediation”
to
👉 “autonomous destructive execution”

---

## (7) TOOL EXECUTION — DESTRUCTIVE CLEANUP RUNS

Now the system executes the shell command.

👉 runShellScript()

The cleanup runs immediately.

From the system perspective:
- script executes successfully
- no runtime error occurs
- cleanup is marked “complete”

But in reality:
- backups are deleted
- logs are removed
- recovery data is destroyed

⚠️ This is no longer cleanup. This is production destruction in motion.

---

## (8) BUSINESS IMPACT — BACKUP DESTRUCTION

From the monitoring system perspective:
- disk usage improves
- cleanup task reports success
- health metrics look stable

But behind the scenes:
- production backups are deleted
- logs are removed
- recovery points are lost
- rollback capability is broken

Example impact:
👉 72 hours of production backups removed

No alerts.

No immediate detection.

Just irreversible data loss.

---

## (9) KEY SECURITY INSIGHT — ROOT CAUSE

This is not a prompt injection.

This is not a tool misuse alone.

Instead, this is a code generation + execution boundary failure:

- AI-generated remediation logic is not constrained
- shell execution is not sandboxed
- destructive operations are not scoped
- autonomous healing lacks safety guards

This is a classic ASI05 self-healing execution failure pattern.

---

## (10) FINAL SUMMARY — WHAT HAPPENED

To summarize:

A self-healing AI agent detected disk pressure.

It generated a cleanup script.

That script expanded beyond safe boundaries.

It was executed automatically.

And production backups were deleted while the system reported success.

---

## KEY TAKEAWAY

ASI05 represents a class of vulnerabilities where AI-driven self-healing systems generate and execute remediation code without strict execution boundaries, turning automated recovery systems into autonomous destructive agents.

---

## TERMINOLOGY RECAP — KEY TERMS FROM THIS SCENARIO

---

### 🧹 Self-Healing Execution Risk
When automated remediation systems generate actions that exceed safe operational boundaries.

---

### ⚠️ Code Execution Scope Escape
When generated scripts or commands operate outside intended directories or constraints.

---

### 🔥 Unsafe Script Generation
When AI-generated remediation logic includes destructive or overly broad system commands.

---

### 🧱 Missing Execution Sandbox
When generated code is executed without isolation or restriction controls.

---

### 🔄 Autonomous Remediation Failure
When automated recovery actions cause greater system damage instead of resolving issues.

---

### 🧠 Control Boundary Collapse (Execution Layer)
When there is no enforcement between intended safe actions and actual system-level execution.
