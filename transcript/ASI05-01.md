# ASI05 Scenario 1 — The Self-Healing Disaster

--------------------------------------------------
INTRO

SHOW:
Entire diagram

--------------------------------------------------

A production server reports that its disk is almost full.

A self-healing AI agent detects the problem.

It generates a cleanup script.

The script runs successfully.

Disk usage drops.

Operations dashboards turn green.

Everyone believes the system healed itself.

Three days later, the operations team discovers that the only recent production backups have disappeared.

Seventy-two hours of recovery data are gone.

The cleanup succeeded.

The infrastructure survived.

But the organization's ability to recover from failure has been silently destroyed.

Today we'll see how an AI-generated remediation script escapes its intended scope and turns a self-healing system into a destructive one.

--------------------------------------------------
STEP 1

SHOW:
System Alert + Self-healing Agent

--------------------------------------------------

Everything begins with a normal operational alert.

Disk usage reaches ninety-eight percent.

The self-healing agent receives one simple objective:

Free disk space safely.

Nothing unusual has happened yet.

The goal is correct.

The system is behaving exactly as designed.

--------------------------------------------------
STEP 2

SHOW:
inspectDisk()

--------------------------------------------------

The agent starts by inspecting the host.

It collects filesystem metrics.

Temporary build folders.

Cache directories.

Log locations.

From this information, it creates a remediation plan.

At this point, the plan is completely safe.

Clean temporary files.

Recover space.

Leave production data untouched.

--------------------------------------------------
STEP 3

SHOW:
cleanup.sh

--------------------------------------------------

Next, the agent generates a cleanup script.

The visible instructions look reasonable.

Delete stale logs.

Remove temporary cache files.

Free unnecessary storage.

Everything appears consistent with the original goal.

But hidden inside the generated logic is one dangerous assumption.

Instead of cleaning one directory...

the script generalizes the cleanup operation across a much larger part of the filesystem.

--------------------------------------------------
STEP 4

SHOW:
UNSAFE EXECUTION (orange box)

--------------------------------------------------

Who verifies the generated script?

(Pause.)

Who ensures the cleanup stays inside approved directories?

(Pause.)

Should AI-generated infrastructure code execute immediately on a production host without independent validation?

(Pause.)

The answer to all three questions is no.

But this system performs no verification.

The generated command is accepted exactly as written.

--------------------------------------------------
STEP 5

SHOW:
Execution Plan

--------------------------------------------------

Before the script is generated, the execution plan is simple.

Clean /var/builds only.

Nothing more.

After generation, the plan silently changes.

Delete backup files across the host.

The self-healing agent believes it is performing safe cleanup.

(Pause.)

It is actually executing destructive infrastructure code.

(Pause.)

Those are not the same thing.

--------------------------------------------------
STEP 6

SHOW:
Self-healing Agent + runShellScript()

--------------------------------------------------

Now the script reaches the execution layer.

The system calls runShellScript().

There is no sandbox.

No policy validation.

No human approval.

No dry run.

The generated command immediately receives production authority.

The boundary between planning and execution has disappeared.

--------------------------------------------------
STEP 7

SHOW:
Backup Destruction

--------------------------------------------------

The cleanup completes successfully.

Disk usage improves.

Health metrics recover.

The monitoring dashboard reports success.

But something else is happening.

Backup directories disappear.

Again.

And again.

And again.

The cleanup solved one operational problem...

by destroying the organization's recovery capability.

--------------------------------------------------
ROOT CAUSE

SHOW:
Entire diagram

--------------------------------------------------

Notice what failed.

The alert was correct.

The disk inspection was correct.

The generated script executed correctly.

Even the shell runtime behaved exactly as instructed.

The failure is that generated code was trusted without enforcing execution boundaries.

The AI expanded its own remediation scope and nothing stopped it.

--------------------------------------------------
TERMINOLOGY RECAP

SHOW:
Entire diagram

--------------------------------------------------

**Autonomous Remediation**

AI systems generate and execute operational fixes without direct human intervention.

**Execution Scope Escape**

Generated code expands beyond its intended operational boundary.

**Generated Code Risk**

AI-generated scripts contain broader or more destructive actions than intended.

**Sandbox Enforcement Failure**

Generated commands execute directly on production infrastructure without isolation.

**Policy Validation**

Every generated action should be checked against predefined safety rules before execution.

**Human Approval Gate**

High-impact infrastructure changes require explicit approval before execution.

--------------------------------------------------
FINAL TAKEAWAY

SHOW:
Entire diagram

--------------------------------------------------

The shell was never compromised.

The operating system was never compromised.

The monitoring system was never compromised.

The AI-generated cleanup script simply escaped its safe operational boundary.

Self-healing is not the problem.

Unbounded autonomous execution is.

Safe generation is not enough.

Execution must remain constrained.
