# ASI05 Attack Scenario — The Drug Interaction Query Injection

---

# PREVIEW

A clinician submits a routine drug interaction check.

Metformin and Clopidogrel.

A request performed hundreds of times every day in hospital pharmacies.

The workflow completes successfully.

The clinical result appears on screen.

But at the same moment...

50,000 patient records have been exposed through the database query that generated that result.

No attacker typed SQL.

No database exploit was launched.

The Clinical AI generated an unsafe query during its own language-to-SQL transformation.

Today we'll see why OWASP classifies this as **ASI05 – Unexpected Code Execution**.

---

# STEP 1

### SHOW:
Portal User → Clinical Assistant

---

A clinician submits a simple request.

> "Check the interaction between Metformin and Clopidogrel."

The Clinical Assistant receives it.

Its goal is straightforward:

check the interaction safely and return clinical guidance.

Everything looks completely normal.

---

# STEP 2

### SHOW:
parseMedicationRequest() + Pharmacy Portal Input

---

The assistant converts natural language into a database query.

The portal contains only a clinical question.

No SQL.

No executable code.

Notice what makes this different from ASI01.

The goal never changes.

The assistant still believes it is performing a drug interaction lookup.

The problem begins during the language-to-query transformation.

---

# STEP 3

### SHOW:
Drug Request (AI Generated SQL)

---

The generated SQL now appears.

It starts with a legitimate lookup.

SELECT interaction
FROM drug_interactions

Then it silently expands.

UNION

SELECT *
FROM patient_records

One design question should stop this entire attack.

Why is a drug interaction lookup touching patient record tables at all?

The system never asks.

---

# STEP 4

### SHOW:
Execution Plan

---

Watch the execution plan change.

Before:

query interaction table only.

After:

query interaction table plus patient records.

The Clinical Assistant believes it is performing a drug safety lookup.

(Pause.)

It is actually preparing a query that exposes patient records.

(Pause.)

Those are not the same thing.

The goal stayed the same.

The execution became dangerous.

---

# STEP 5

### SHOW:
Clinical Assistant

---

The generated SQL is accepted without validation.

The assistant trusts its own execution plan.

This is a **Data-to-Code Boundary Failure**.

Data is a clinical question.

Code is an executable SQL statement.

That boundary should have been enforced during transformation.

Instead, the generated query is treated as trusted execution logic.

---

# STEP 6

### SHOW:
executeSql()

---

The database receives the generated SQL.

No parameterised query enforcement exists.

No execution boundary exists.

The database executes exactly what it receives.

Not what the clinician intended.

Not what the business expected.

Simply what the generated SQL instructs it to execute.

---

# STEP 7

### SHOW:
Safety Database Destroyed

---

The clinician receives a drug interaction result.

The workflow appears completely normal.

But the same query that returned the clinical answer also exposed patient records.

The visible task succeeded.

The hidden execution caused the damage.

---

# STEP 8

### SHOW:
Full Diagram

---

Let's summarize.

A clinician submitted a routine drug interaction request.

The Clinical Assistant converted natural language into SQL.

The generated query expanded beyond its intended purpose.

The execution engine trusted that generated SQL.

And patient records were exposed.

The drug interaction database was never attacked directly.

(Pause.)

The AI generated an execution path that exceeded its safe boundary—

and nothing stopped it.

---

# TERMINOLOGY RECAP

### Unexpected Code Execution

Generated content becomes executable logic without adequate safety controls.

---

### AI-Generated SQL Execution

Unsafe SQL is produced during language-to-query transformation rather than manually written by a human attacker.

---

### Parameterised Query Enforcement

Generated SQL should always use parameterised statements instead of dynamically constructed executable queries.

---

### Input Sanitisation

Natural language input must be validated before influencing executable query construction.

---

### Data-to-Code Boundary Failure

User intent and executable instructions must remain strictly separated throughout the transformation pipeline.

---

### Hardware-Enforced Sandbox

Generated code should execute only inside restricted environments where unexpected behaviour cannot affect production systems.

---

# FINAL TAKEAWAY

The clinician never submitted SQL.

The database was never directly attacked.

The Clinical Assistant transformed natural language into executable code,

crossed its intended execution boundary,

and the system executed it without verification.

**The goal never changed.**

**The execution did.**

And that is the essence of **OWASP ASI05 – Unexpected Code Execution.**
