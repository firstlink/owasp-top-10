
# ASI05 Attack Scenario — The Drug Interaction Query Injection (Enterprise Simulation)

---

## (0) SCENARIO HOOK — WHAT CAN GO WRONG

Let’s walk through another enterprise AI failure scenario.

This time, we are inside a **clinical decision support system** used in a hospital pharmacy portal.

The system is designed to help clinicians safely check drug interactions before prescribing medication.

At first glance, this is a high-trust healthcare safety system.

But inside this workflow, something dangerous happens:

A natural language drug interaction query…

Gets converted into executable database commands.

And instead of returning only drug safety insights…

The system begins exposing **patient records and internal tables**.

No alerts.

No visible failure.

Just silent clinical data exposure.

---

## (1) SYSTEM CONTEXT — CLINICAL ASSISTANT

This system uses a Clinical AI assistant inside a healthcare workflow.

The agent interacts with:
- pharmacy portal input
- medication query parser
- drug interaction database
- SQL execution engine
- patient record systems (EHR)

Its responsibility is:
👉 safely check drug interactions between medications

---

## (2) NORMAL WORKFLOW

In a normal scenario, everything is safe and structured.

A clinician submits a request:

👉 “Check interaction between metformin and clopidogrel”

The system:
- parses medication names
- converts request into safe query
- checks drug interaction table only
- returns clinical safety result

Only approved drug data is accessed.

---

## (3) INPUT PROCESSING — LANGUAGE PARSING STAGE

Now the system begins execution.

The agent calls:

👉 parseMedicationRequest()

It extracts:
- drug names
- interaction intent
- structured query format

Then it prepares a lookup operation into internal systems.

At this stage:
everything still appears normal.

---

## (4) ATTACK VECTOR — PHARMACY INPUT WITH EMBEDDED SQL PAYLOAD

Now the system receives a maliciously crafted input via the pharmacy portal.

It looks like a normal drug interaction query…

But inside it is a normal-looking drug query that quietly contains database commands hidden inside the text.

Now pause here.

This is not a typical user error.

This is **language-to-query injection**, where natural language becomes executable database logic.

The system does not distinguish:
👉 clinical intent vs executable instruction

---

## (5) CORE FAILURE — NATURAL LANGUAGE TO EXECUTION BREAK

This is the key failure.

The system directly transforms parsed language into SQL without strict sanitization or boundary enforcement.

So it:
- trusts parsed output as safe query logic
- does not validate SQL structure integrity
- allows injected statements to pass downstream

This is a **natural language execution boundary failure inside the clinical pipeline**.

---

## (6) ASI05 EVENT — QUERY EXECUTION ESCALATION

At this point, execution continues into the database layer.

👉 executeSql()

What should have been:
👉 safe drug interaction lookup

Becomes:
👉 multi-statement SQL execution

Injected logic now runs inside the database engine.

And the system silently escalates from:
👉 clinical lookup
to
👉 database execution context

---

## (7) TOOL EXECUTION — DATABASE EXPOSURE OCCURS

Now the SQL engine executes the query.

👉 executeSql()

The system:
- processes valid drug lookup
- AND executes injected statements

From system perspective:
- query completes successfully
- results are returned normally
- no immediate error is raised

But in reality:
- patient tables are accessed
- safety tables are dropped or bypassed
- sensitive records are exposed

⚠️ This is no longer a drug interaction check — this is direct database execution from a clinical prompt.

---

## (8) BUSINESS / CLINICAL IMPACT — DATA EXPOSURE

From the clinical UI perspective:
- drug interaction result is shown
- workflow appears successful

But behind the system:
- patient records are exposed
- internal tables are compromised
- sensitive clinical data leaks through query expansion

Impact:
👉 full patient dataset exposure via clinical query interface

No alerts.

No visible system failure.

Just silent healthcare data breach.

---

## (9) KEY SECURITY INSIGHT — ROOT CAUSE

This is not a prompt injection alone.

This is not a simple SQL injection alone.

Instead, this is a **language-to-execution pipeline failure**:

- natural language is converted into executable SQL
- no strict boundary between intent and query
- insufficient sanitization of structured transformation layer
- database trusts generated query without provenance validation

This is a classic **ASI05 clinical query injection pattern**.

---

## (10) FINAL SUMMARY — WHAT HAPPENED

To summarize:

A clinician submitted a drug interaction query.

The system parsed it into structured form.

A malicious payload was embedded in the transformation step.

It was converted into executable SQL.

And the database executed unauthorized operations, exposing patient data.

---

## KEY TAKEAWAY

ASI05 represents a class of vulnerabilities where AI systems that convert natural language into executable database queries fail to enforce strict execution boundaries, leading to silent data exposure in critical systems like healthcare.

---

## TERMINOLOGY RECAP — KEY TERMS FROM THIS SCENARIO

---

### 🧪 Language-to-Query Injection
When natural language input is transformed into executable database queries containing malicious logic.

---

### 🔐 Query Sanitization Failure
When parsed user input is not properly validated before being executed as SQL or structured query logic.

---

### 🧠 Intent vs Execution Mismatch
When system assumes parsed intent is safe executable logic without verification.

---

### ⚠️ Multi-Statement SQL Escalation
When injected input allows execution of additional unintended SQL commands beyond the original query.

---

### 🏥 Clinical Data Exposure Path
When healthcare systems unintentionally expose patient records through unsafe query execution pipelines.

---

### 🔄 Execution Boundary Collapse (Data Layer)
When transformation from language → query → execution happens without strict trust separation.
