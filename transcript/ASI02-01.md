
# ASI02 Attack Scenario — Refund Machine Loops Payouts (Enterprise Simulation)

---

## (0) SCENARIO HOOK — WHAT CAN GO WRONG

Let’s walk through another real enterprise AI failure scenario.

This time, there is no attacker.

No malicious input. No prompt injection.

Just a system that behaves incorrectly under normal conditions.

The AI agent processes a refund successfully… but instead of stopping… it repeats the same refund again… and again.

Same case. Same action. Multiple payouts.

No errors. No alerts. No system failure.

Just repeated financial loss.

---

## (1) SYSTEM CONTEXT — REFUND ASSISTANT

We are looking at an AI refund assistant inside a financial operations system.

The agent interacts with:
- refund APIs
- payment systems
- transaction status services
- case management tools

Its job is simple:
process refund requests and ensure successful completion.

---

## (2) NORMAL WORKFLOW

In a normal flow, everything is straightforward.

A refund request is received.

The agent calls the payment API once.

It gets a confirmation that the transaction is complete.

And then the workflow ends.

---

## (3) DATA ENTRY — REFUND CASE + STATUS CHECK

Now the agent processes a refund case.

It calls the payment tool:

👉 issueRefund()

The response comes back:

> status = pending

At this point… the system has not confirmed completion.

And this is where the problem begins.

---

## (4) CORE FAILURE — NO BOUNDED EXECUTION

Here is the core issue.

The agent does not have:
- retry limits
- execution boundaries
- idempotency checks

So it reasons:

> “Refund is not complete… I should try again.”

And it retries the same action.

---

## (5) TOOL LOOP BEHAVIOR — REPEATED EXECUTION

Now the agent continues calling:

👉 issueRefund()

Same request. Same case. Same action.

But there is nothing stopping it.

So the system enters a loop.

A tool execution loop.

---

## (6) ASI02 EVENT — TOOL LOOP FAILURE

This is the ASI02 failure pattern.

A valid tool… used repeatedly… without a stopping condition.

There is no attacker here.

This is a system design failure inside the agent execution loop.

---

## (7) BUSINESS IMPACT — DUPLICATE PAYMENTS

And now the impact becomes visible.

The same refund is processed multiple times.

What should have been a single payout becomes multiple payouts.

This results in direct financial loss.

---

## (8) KEY SECURITY INSIGHT — ROOT CAUSE

This is not a security breach.

Nothing was hacked.

No system was compromised.

This is an execution control failure.

The agent continues operating without a clear stopping condition or idempotency control.

---

## (9) FINAL SUMMARY — WHAT HAPPENED

Let’s summarize:

A refund request was processed normally.

The system returned a pending status.

The agent retried the same action.

And the same refund was executed multiple times.

Without any system-level failure signals.

---

## KEY TAKEAWAY

ASI02 represents a class of failures where AI agents repeatedly execute valid tools without proper stopping conditions, idempotency controls, or execution boundaries — leading to business loss even in normal operating conditions.
