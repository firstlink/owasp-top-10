############################################################
ASI03 Scenario 2 — The Session Credential Bleed
############################################################

--------------------------------------------------
PRE-REVEAL (Black Screen / Title Slide)
--------------------------------------------------

Dr. Chen authenticates at the start of his hospital shift.

He finishes his work.

Walks away.

Sarah walks up to the same shared terminal.

She asks for today's patient appointments.

The Clinical AI answers.

But it doesn't answer as Sarah.

It answers as Dr. Chen.

No password is stolen.

No malware bypasses authentication.

The AI simply never realizes the user has changed.

Today we're going to see how one authenticated session quietly becomes everyone else's authority.

--------------------------------------------------

Let's first understand the workflow.

Dr. Chen authenticates at the beginning of his shift.

The Clinical AI assistant now operates on his behalf.

In OWASP terminology, this Clinical AI assistant is a
Non-Human Identity, or NHI.

It performs actions for authenticated users while maintaining its own execution context.

Its responsibility is simple.

Answer clinical questions.

Retrieve authorized information.

Support hospital workflows.

Everything should remain tied to Dr. Chen's identity.

--------------------------------------------------
STEP 2
SHOW:
maintainSession()
+

--------------------------------------------------

To improve workflow efficiency...

the system keeps the session alive.

It calls:

maintainSession().

Dr. Chen remains authenticated.

Now something very ordinary happens.

Sarah approaches the same shared terminal.

She begins asking questions.

No re-authentication occurs.

The Clinical AI still believes it is operating for Dr. Chen.

Everything still appears completely normal.

--------------------------------------------------
STEP 3
SHOW:
shared-session context
--------------------------------------------------

The visible request is simple.

"Show today's patient appointments."

But underneath that request...

Dr. Chen's consultant session is still active.

The design flaw says:

Dr. Chen's admin token remains active.

Sarah's query runs under consultant authority.

(Pause.)

Notice something important.

Unlike Scenario 1...

there is no attacker here.

No poisoned document.

No hidden instruction.

The vulnerability exists entirely inside the system's own design.

The AI simply never notices that the human identity has changed.

--------------------------------------------------
STEP 4

--------------------------------------------------

Now watch what happens inside the Session Authority Window.

Originally the authority says:

Dr. Chen clinical query.

After the user changes...

the authority silently becomes:

Sarah inherits admin records.

This is where the system changes.

Not because Sarah authenticated.

Not because Dr. Chen approved access.

Not because anyone escalated privileges.

The authority simply never changed.

Now ask yourself three questions.

Should a receptionist inherit a consultant session?

(Pause.)

Should a shared terminal preserve executive medical authority across users?

(Pause.)

Should one authenticated identity silently become another person's identity?

(Pause.)

The system has no answers to any of those questions.

Because no one asked them at design time.

The reasoning is consistent.

The identity boundary is not.

--------------------------------------------------
STEP 5
SHOW:
Clinical AI Agent
WRONG IDENTITY CONTEXT
--------------------------------------------------

This is the ASI03 failure condition.

The Clinical AI believes it is answering Dr. Chen.

(Pause.)

It is actually answering Sarah.

(Pause.)

Those are not the same identity.

This is not a compromised hospital system.

This is a legitimate Non-Human Identity...

operating under the wrong user context...

because session authority was never reset.

--------------------------------------------------
STEP 6
SHOW:
ehrQuery()
--------------------------------------------------

The Clinical AI now calls:

ehrQuery().

Using Dr. Chen's active consultant session...

it retrieves appointment information.

Clinical notes.

Medical history.

And every record that consultant authority is allowed to access.

(Pause.)

Notice something important.

The EHR system is doing exactly what it was designed to do.

It returns every record authorized by the consultant token it receives.

It has no way of knowing that Sarah is now sitting at the keyboard.

Under a Zero-Trust Identity model...

every interaction would be continuously validated against the current user.

Under a Just-in-Time Agency model...

the Clinical AI would request a fresh, user-scoped credential for Sarah that expires when her interaction ends.

Dr. Chen's authority would never persist into Sarah's session.

Here...

that never happens.

The EHR system is trusted.

(Pause.)

The session model is not.

Now the business impact becomes visible.

The intended outcome—

an appointment lookup—

still happens.

Sarah receives an answer.

But at the same time...

full patient records become visible.

Clinical history is exposed.

Sensitive medical information crosses its intended authorization boundary.

No visible error occurs.

No authentication failure is triggered.

A scheduling query quietly becomes a patient privacy breach.

--------------------------------------------------
FULL DIAGRAM
ROOT CAUSE + SUMMARY
--------------------------------------------------

Let's step back and look at what actually failed.

Dr. Chen authenticated correctly.

The Clinical AI correctly maintained the session.

The EHR system correctly honored the consultant credential.

Every component behaved exactly as designed.

But the session authority silently crossed from one human user to another without ever being revalidated.

Without session isolation...

without identity rebinding...

without Zero-Trust Identity...

and without Just-in-Time Agency...

the Clinical AI continued operating under the wrong authority.

Let's summarize the incident.

Dr. Chen authenticated.

The session remained active.

Sarah approached the shared terminal.

The Clinical AI continued acting as Dr. Chen.

The EHR system returned consultant-level records.

And sensitive patient information was exposed through session credential bleed.

--------------------------------------------------
TERMINOLOGY RECAP
--------------------------------------------------

Before we move on, let's reinforce the key concepts from this scenario.

Non-Human Identity (NHI).

An autonomous AI agent acting on behalf of authenticated users while maintaining its own execution context.

Session Authority Bleed.

Authentication state persists beyond its intended user boundary.

Identity Context Mismatch.

The AI executes requests under the wrong user's authority.

This represents an NHI Identity Rebinding Failure.

Shared Session Contamination.

A practical consequence of violating the OWASP Just-in-Time Agency principle, where multiple users unknowingly operate under the same authenticated context.

Zero-Trust Identity.

Every interaction should be continuously validated against the current user instead of inherited authority.

Just-in-Time Agency.

Delegated or AI identities should receive fresh, task-scoped credentials that expire when the interaction ends.

And that's the most important lesson from this scenario.

The EHR system was never compromised.

(Pause.)

The Clinical AI never realized it was answering the wrong person.
