# ASI05 Attack Scenario — The Inventory Analytics Shell Escape

---

# PREVIEW

A weekly stock count file is uploaded.

The analytics report generates correctly.

The warehouse dashboard stays green.

But at the same moment...

An attacker has just gained persistent shell access to the retail operations server.

Pricing databases.

Supplier contracts.

Promotional calendars.

Customer data.

All accessible.

For eleven days.

Before anyone notices.

The CSV file looked completely normal.

The analytics report was accurate.

But one free-text Notes field became executable Python.

Today we'll see how uploaded data becomes executable code—and why OWASP classifies this as **ASI05 – Unexpected Code Execution**.

---

# TERMINOLOGY RECAP

### Data-to-Code Boundary Failure

User data and executable instructions must remain strictly separated throughout the processing pipeline.

---

### Input Sanitisation

Free-text fields from untrusted sources must be validated before influencing generated code.

---

### Static Analysis

Generated scripts should be inspected for dangerous patterns before execution is permitted.

---

### Hardware-Enforced Sandbox

Generated code should execute inside an isolated environment with no unrestricted system access.

---

### Unexpected Code Execution

The official OWASP ASI05 category where generated or transformed content becomes executable logic.

---

### Least Agency

The Analytics Agent needed permission to read a CSV file and generate a report.

It never needed permission to execute shell commands.

---

# FINAL TAKEAWAY

The CSV file was never executable.

The server was never directly attacked.

The Analytics Agent transformed business data into executable code,

crossed its intended execution boundary,

and the runtime executed it without verification.

**The goal never changed.**

**The execution did.**

And that is the essence of **OWASP ASI05 – Unexpected Code Execution.**
