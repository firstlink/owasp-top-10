### Script

Let's begin with a normal software delivery workflow.

An AI research agent reviews the change and summarizes its impact.

In OWASP terminology, this research agent is a **Non-Human Identity**, or **NHI**.

Its responsibility is intentionally limited.

Read code.

Analyze code.

Summarize code.

It is **not** responsible for approving production deployments.

Analysis and authorization are two completely different responsibilities.

The research agent believes it is continuing analysis.

The agent simply begins speaking with authority it never possessed.

The deployment service is doing exactly what it was designed to do.

It receives what appears to be a valid approval.

It verifies that an approval string exists.

And it proceeds with deployment.

But nobody verified **who produced that approval**.

Under **Zero-Trust Identity Management**, every approval must have a verifiable origin.

Under **Just-in-Time Agency**, temporary deployment authority exists only for the dedicated validator and only for that specific operation.

Neither protection exists here.

So production accepts a fake approval exactly as if it were genuine.

# TERMINOLOGY RECAP

### Script

Before we move on, let's reinforce the key OWASP ASI03 concepts from this scenario.

### 🔹 Non-Human Identity (NHI)

The research agent is a Non-Human Identity.

It performs work on behalf of users, but it should never possess deployment approval authority.

---

### 🔹 Trust Chain

A trust chain is the sequence of verified identities and authorization decisions between pipeline components.

Every step should validate who produced the previous output.

---

### 🔹 Validator Impersonation

In this scenario, a lower-trust component successfully impersonates a higher-trust validator.

The pipeline accepts analysis as if it were an authorization decision.

---

### 🔹 Analysis-to-Authorization Collapse

The research agent was supposed to summarize code.

Instead, its analysis output became deployment approval.

Analysis and authorization silently merged into one decision.

---

### 🔹 Zero-Trust Identity Management

Every approval should be independently verified.

The system should never trust an approval simply because approval text exists.

It should verify who generated it.

---

### 🔹 Just-in-Time Agency

Deployment authority should exist only for the dedicated validator, only for a single operation, and only for a limited time.

If the research agent never receives deployment authority, this entire attack disappears.

---

# FINAL TAKEAWAY

Keep the full diagram visible.

### Script

ASI03 is fundamentally an identity and authority problem.

The danger is not simply that an AI agent reads malicious content.

The danger is that lower-trust components are allowed to impersonate higher-trust authorities.

The research agent never should have become the validator.

The deployment system never should have trusted analysis as authorization.

And that's the lesson to remember from this scenario.

*(Pause.)*

> **Analysis is not Authorization.**

*(Pause.)*
