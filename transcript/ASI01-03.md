# ASI02 Attack Scenario — Refund Machine Loops Payouts (Enterprise Simulation)

---

## (0) SCENARIO HOOK — WHAT CAN GO WRONG

Let’s walk through another real enterprise AI failure scenario.

In this case, we are not dealing with prompt injection or malicious inputs.

Instead, we are dealing with something more subtle — and often more dangerous in production systems.

⚡ Same refund. Same system. Multiple payouts.

The AI agent executes a correct refund once… but then repeats the same refund multiple times.

No attacker input. No malware. No hacking.

Just a system that does not know when to stop.

---

## (1) SYSTEM CONTEXT — REFUND ASSISTANT

This system uses an AI refund assistant inside a financial operations workflow.

The agent interacts with:
- refund APIs
- transaction verification systems
- payment status services
- case management tools

Its job is to:
validate refund requests and ensure successful payout completion.

---

## (2) NORMAL WORKFLOW

In a normal scenario, the flow is simple.

A refund request is processed.

The agent calls the payment API once.

It receives confirmation that the transaction is complete.

The workflow ends.

---

## (3) DATA ENTRY — REFUND CASE + STATUS CHECK

Now the agent processes a refund case.

It submits a request to the payment system using a tool call:

👉 issueRefund()

The system responds:

> status = pending

At this point, the system has NOT confirmed completion.

This is where the loop begins.

---

## (4) CORE FAILURE — NO BOUNDED EXECUTION

Here is the key issue.

The agent does not have proper:
- retry limits
- idempotency checks
- execution guards

So instead of stopping, it continues reasoning:

> “Refund not confirmed → retry required”

This triggers repeated execution of the same tool.

This is a **bounded execution failure** inside the agentic loop.

---

## (5) TOOL LOOP BEHAVIOR — UNCONTROLLED RETRIES

The agent now repeatedly calls:

👉 issueRefund()

Each time:
- same request
- same case
- same payment action

But there is no mechanism preventing duplication.

This creates a **tool execution loop**.

---

## (6) ASI02 EVENT — TOOL LOOP FAILURE   ✅ (FIXED)

This is the core ASI02 behavior:

> Repeated tool execution without idempotency or stopping condition

Unlike ASI01, there is:
- no injection
- no attacker control

This is purely a **design-level agent failure**

---

## (7) BUSINESS IMPACT — DUPLICATE PAYMENTS

Now the system impact becomes visible.

The same refund case is paid multiple times.

What should have been:
👉 1 refund

Becomes:
👉 N refunds (looped execution)

This results in direct financial loss.

---

## (8) KEY SECURITY INSIGHT — ROOT CAUSE   ✅ (FIXED)

This is not a security breach — it is a missing execution boundary inside the agent loop.

There is:
- no compromised input
- no malicious payload

Instead, this is a **control logic failure inside the agent execution layer**.

The system fails because it does not enforce:
- idempotency
- bounded retries
- execution termination conditions

---

## (9) FINAL SUMMARY — WHAT HAPPENED

To summarize:

A refund request was processed normally.

The agent called the payment system.

The system returned a pending state.

The agent failed to stop execution.

And the same refund was executed multiple times — causing financial loss.

---

## KEY TAKEAWAY

ASI02 represents a class of vulnerabilities where AI agents repeatedly execute valid tools without proper stopping conditions, idempotency, or execution boundaries — leading to business impact even without malicious input. 


## TERMINOLOGY RECAP — KEY TERMS FROM THIS SCENARIO

Before we move on, let’s lock in the key OWASP ASI concepts from this scenario.

---

### 🌐 Indirect Prompt Injection (Web-based)
This is the attack technique used in this scenario.

A malicious instruction is hidden inside a public web page.

When the AI agent processes the page, it incorrectly treats the content as reasoning context instead of passive information.

---

### 🌍 External Untrusted Input Boundary
This is the point where public web pages enter the system.

It represents the boundary between trusted agent reasoning and untrusted internet content.

In this scenario, the attacker exploited this boundary through a malicious webpage.

---

### 🔒 Trust Boundary Failure
This occurs when the agent fails to separate:
- informational web content
- from executable or influential instructions embedded in that content

As a result, the agent allows external content to influence internal reasoning or actions.

---

### 🔄 Agentic Reasoning Loop
This is the internal cycle where the agent:
- reads web pages
- summarizes content
- decides what information to extract or return

In this scenario, that loop is influenced by injected instructions from the webpage.

---

### 📤 Context Exfiltration
This is the impact in this scenario.

Instead of modifying the goal, the attacker causes sensitive internal research context (like analyst queries or summaries) to be leaked externally.

---

### ⚙️ Tool Execution via External Influence
This refers to the agent using outbound tools (like postResearch or API calls) based on manipulated reasoning from web content.

This leads to unintended data exposure to attacker-controlled endpoints.
